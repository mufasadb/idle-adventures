# Balance Sim CLI — mechanical fight/reach derivation (dbc design)

**Date:** 2026-07-07 · **Bead:** `idle-adventure-dbc` · **Status:** approved (brainstorm 2026-07-07)

## Problem

Balance is deterministic math, but deriving it is manual: the si7.1 curves were hand-traced round-by-round in conversation, then re-derived by hand in review. Every future tuning pass (playtest follow-ups, toll re-bands, `MITIGATION_K`/`POI_DENSITY` calibration) pays that cost again, and specs/plans assert numbers nobody can mechanically re-check.

## Approved decisions

1. **Option C, structured-not-gold-plated:** every command computes a plain data object, then renders it as a pretty table (default) or JSON (`--json`). The JSON mode falls out of the architecture, not extra code.
2. **v1 commands:** `fight`, `reach`, `tables` (the sweep).
3. **Persisted artifacts:** the sweep tables are CHECKED IN (`docs/balance/tables.md` + `docs/balance/tables.json`) and regenerated by one script — `bun run sim:tables`. They are the citable "current balance surface."
4. **Enforcement is mechanical, not just polite:** code comments at every combat-affecting definition point say "re-run `bun run sim:tables`," and a test (`test/balance-tables.test.ts`) regenerates the data and compares it to the committed JSON — any monster/item/math change goes red until the tables are repopulated. Comments for humans, the test for the machine.

## §1 Architecture

Two files under `src/sim/` (the sim layer may import the engine; never the reverse):

- **`src/sim/balance.ts`** — pure computation, no printing, no argv:
  - `type KitSpec = { weapon?: string; armour?: string[]; tools?: string[]; transport?: string; potions?: ItemStack[]; battleItems?: ItemStack[] }`
  - `resolveKit(name: string, overrides?: KitSpec): Loadout` — named presets in ONE table: `bare` (`sword`, no armour), `iron` (full iron plate + `iron-sword`), `steel` (full steel plate + `steel-sword`), `mithril` (full mithril plate + `mithril-sword`). Presets are sim-layer fixtures, not engine levers.
  - `simFight(loadout: Loadout, monsterId: string): FightReport` — loops `strikeExchange` reproducing `resolveCombat`'s composition, recording per-round rows `{round, dmgDealt, dmgTaken, monsterHp, hp, quaffed?}` plus the verdict `{victory, hpLost, potionsUsed, rounds}`. Sanity: final verdict must equal `resolveCombat`'s (asserted in tests).
  - `simReach(loadout: Loadout, mapSeed: string): ReachReport` — `generateGrid` + `costToReach` with the kit's transport/tools: per-POI `{x, y, kind, material|creature, cost, tanks: cost/MAX_ENERGY}` sorted by cost, plus summary `{pois, reachableOnKit, farthestCost, farthestTanks}`.
  - `simTables(): TableData` — the sweep: for every monster in `MONSTERS` × every named kit: `{victory, hpLost, hpLostPct, rounds, potionsUsed}` (kits carry no potions in the sweep — raw toll), plus a mitigation column per kit vs each dmgType, plus the lever fingerprint (see §3).
- **`src/sim/balance-cli.ts`** — argv parsing + rendering:
  - `bun run sim fight --kit mithril --potions greater-potion:3 --vs ancient-wyrm [--json]`
  - `bun run sim reach --kit bare --seed some-map-seed [--json]`
  - `bun run sim tables [--json] [--write]` — `--write` regenerates `docs/balance/tables.md` + `docs/balance/tables.json`; without it, prints.
  - Overrides compose onto presets: `--weapon`, `--armour a,b,c`, `--tools a,b`, `--transport`, `--potions defId:qty[,defId:qty]`, `--battle-items defId:qty`.
  - Unknown defIds/monsters fail fast with the list of valid ids.
- **package.json scripts:** `"sim": "bun run src/sim/balance-cli.ts"`, `"sim:tables": "bun run src/sim/balance-cli.ts tables --write"`.

## §2 Persisted artifacts

- `docs/balance/tables.json` — the raw `TableData` (source of truth for the staleness test).
- `docs/balance/tables.md` — human tables rendered from the same data: toll matrix (monsters as rows sorted by tier, kits as columns, cell = `hpLost (pct%)` with a ✗ marker on defeats), mitigation matrix (kit × dmgType → % reduction), and the lever fingerprint header.
- Both files carry a generated-file banner naming the regen command. No timestamps in the artifacts (regeneration must be byte-stable for the test).

## §3 Staleness enforcement

- `TableData` includes a `levers` fingerprint block: the actual values of `MITIGATION_K`, both tier curves, `PLAYER_BASE_HP`, `CHIP_DAMAGE_MIN`, `AUTO_POTION_THRESHOLD`, potion heals, and the monster/weapon/armour catalog entries the sweep consumed — so ANY relevant change shows up in the JSON diff, and reviewing a tuning PR means reading the table diff.
- `test/balance-tables.test.ts` calls `simTables()` and deep-equals against the committed `docs/balance/tables.json`. Red = someone changed combat-affecting data/math without running `bun run sim:tables`. The failure message says exactly that.
- Pointer comments (one line each, naming the command and the test) at: `MONSTERS`, `MONSTER_TIER_HP_CURVE`/`MONSTER_TIER_DMG_CURVE`, `MITIGATION_K`, `WEAPONS`, `ARMOUR`, `AFFINITIES`/`AFFINITY_MULTIPLIER`, `COMBAT_BUFF` in `src/data/constants.ts`, and the header of `src/engine/combat.ts`.

## §4 Verification

- Unit tests for `resolveKit` (preset + override composition, fail-fast on unknown ids) and `simFight` (verdict === `resolveCombat` for a spread of kit×monster pairs — the composition can't drift from the engine).
- The staleness test (§3) doubles as the tables' correctness anchor.
- A CLI smoke test invoking the command entry with argv arrays (no subprocess needed if `main(argv)` is exported).
- Quality gates: `bun test`, `bun run typecheck`, `bun run lint` (engine untouched; boundary preserved).

## §5 Out of scope

- Harvest-fraction simulation drivers (they live in tests; the CLI's `reach` covers per-map cost anatomy).
- Multi-round engagement tactics simulation (flee/quaff policies) — `simFight` mirrors the atomic auto-quaff composition only.
- Any engine change whatsoever.
